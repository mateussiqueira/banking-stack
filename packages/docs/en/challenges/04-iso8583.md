# Challenge 04 — ISO 8583 Simulator

**What is it:** A simulator for ISO 8583 — the binary protocol that powers credit and debit card transactions.

**Why it matters:** Every time you tap your card, an ISO 8583 message is sent. It's been around since 1987 and still runs the world's card networks.

## The problem

ISO 8583 is a binary protocol, not JSON or XML. Messages look like this:

```
0200 723C 0001 0800 0000 0000 0000 0001
0000 0000 0000 0000 0000 0000 0000 0000
...
```

Each bit in the header tells you which fields are present. Each field has its own format (numeric, alphanumeric, binary).

This is how card networks have worked for 40 years. It's fast, compact, and battle-tested.

## The message types

| Code | Name | Description |
|------|------|-------------|
| 0100 | Authorization Request | "Can I charge this card?" |
| 0110 | Authorization Response | "Yes/No" |
| 0200 | Financial Transaction | "I'm charging this card" |
| 0210 | Financial Response | "Done" |
| 0400 | Reversal Request | "Undo that transaction" |
| 0410 | Reversal Response | "Undone" |

## Why TypeScript

ISO 8583 is complex. The binary parsing, the field definitions, the bitmap handling — it's a lot of code.

TypeScript is better here because:
- Easier to debug binary data
- Better libraries for buffer manipulation
- Faster prototyping
- The bottleneck is network I/O, not CPU

## How it works

```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│   Acquirer   │───▶│   ISO 8583   │───▶│   Issuer     │
│   (POS)      │    │   Simulator  │    │   (Bank)     │
└──────────────┘    └──────────────┘    └──────────────┘
```

The simulator can act as:
- **Acquirer** — receives card transactions
- **Issuer** — authorizes or declines transactions
- **Admin** — manages test cards
